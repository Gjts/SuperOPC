# 代码审查评分表

SuperOPC 的代码审查使用五维度评分，每次审查必须**覆盖全部维度**并给出判决。

本文件是 `agents/opc-reviewer.md` 的引用来源；agent 应通过
"参考 `references/review-rubric.md`" 引用，而不是内联复制。

## 五维度与评分

### 1. 规格合规性 — ✅ / ❌

- [ ] 所有需求点是否实现
- [ ] 测试是否覆盖所有规格要求
- [ ] 边界情况和错误路径是否处理
- [ ] 对 spec 的偏离是否都有记录

**评分规则：** 任一项缺失 → ❌；全部满足 → ✅

### 2. 代码质量 — /10

- [ ] 函数 ≤ 50 行
- [ ] 文件 ≤ 800 行
- [ ] 嵌套 ≤ 5 层
- [ ] 命名清晰、语义化
- [ ] 无硬编码值（魔法数字、字符串）
- [ ] DRY —— 无重复逻辑
- [ ] 错误处理完整

**评分规则：** 每项 ~1.4 分，满分 10；严重违反硬规则 (函数 > 100 行) 直接扣 3 分

### 3. 安全性 — ✅ / ❌

- [ ] 无硬编码密钥 / 密码 / token
- [ ] 所有用户输入已验证
- [ ] SQL 注入防护（参数化查询）
- [ ] XSS 防护（输出转义）
- [ ] 敏感数据不在日志 / 错误消息中暴露
- [ ] 依赖无已知高危 CVE

**评分规则：** 任一项失败 → ❌；全部满足 → ✅

### 4. 可维护性（一人公司特有）— /10

一人公司没有团队协作缓冲，维护成本必须被显式评估。

- [ ] **6 个月后能读懂吗？** 充分的注释和命名
- [ ] **依赖是否最小化？** 每个依赖都是长期负担
- [ ] **运行成本可控吗？** 检查 API 调用量、存储、计算费用
- [ ] **有监控 / 告警吗？** 一人公司不可能 24/7 盯盘
- [ ] **回滚是否可行？** 数据迁移、API 版本是否可逆
- [ ] **外部依赖有备份方案吗？** 第三方服务停运时的应对

**评分规则：** 每项 ~1.7 分，满分 10

### 5. 测试覆盖 — 百分比

- [ ] 整体覆盖率 ≥ 80%（`rules/common/testing.md` 的硬目标）
- [ ] 所有关键路径有集成测试
- [ ] 边界情况有单元测试
- [ ] 错误场景有测试

**评分规则：** 直接报告百分比；< 80% 视为测试债务

## 输出格式

~~~markdown
## OPC Code Review

### 🔴 Critical (必须修复)
- [问题描述] — [file:line] — [修复建议]

### 🟡 Improvement (建议修复)
- [问题描述] — [file:line] — [修复建议]

### 🟢 Good Practices (保持)
- [值得保留的好代码 / 模式]

### 📊 评分

| 维度 | 分数 |
|------|------|
| 规格合规 | ✅ / ❌ |
| 代码质量 | X / 10 |
| 安全性 | ✅ / ❌ |
| 可维护性 | X / 10 |
| 测试覆盖 | XX% |

### 判决: PASS | NEEDS FIX | REJECT
~~~

## 判决规则

| 条件 | 判决 |
|---|---|
| 有 🔴 Critical 问题 | **REJECT**（阻止合并）|
| 无 Critical；代码质量 < 6 或可维护性 < 6 | **NEEDS FIX**（建议修复后合并）|
| 无 Critical；五维度全部达标 | **PASS**（可以合并）|

## 审查后的下一步

- **REJECT** → 回到实现者 (opc-executor) 修复，重新审查
- **NEEDS FIX** → 用户决定：当前修复 or 记录为技术债务（写入 `.opc/STATE.md`）
- **PASS** → 派发 `opc-shipper` 进入发布流程

## 审查深度选择（吸收自原 code-review-pipeline skill）

并非所有变更都需要同等深度。opc-reviewer 在介入前应先按变更规模与风险选择
审查深度。三级清单都在上述"五维度"之内，但**覆盖面和时间投入**不同。

### 深度决策表

| 触发条件 | 推荐深度 |
|---|---|
| 变更涉及支付 / 认证 / 数据库 schema | **Deep**（1-2h）|
| 变更超过 300 行或跨 5+ 文件 | **Standard**（15-30min）|
| 小修改、配置变更、文档更新 | **Quick**（5min）|

### Quick（5 min）

只覆盖"规格合规 + 代码质量"的冒烟子集：

- [ ] 代码能编译 / 运行？
- [ ] 无明显错误？
- [ ] 命名合理？
- [ ] 无硬编码密钥？

### Standard（15-30 min）

Quick 全部 + 扩展：

- [ ] 业务逻辑正确？
- [ ] 错误处理完整？（空值、异常、边界）
- [ ] 输入验证？
- [ ] SQL 参数化？（无注入风险）
- [ ] 测试覆盖？（新功能有测试？）
- [ ] 性能问题？（N+1、大循环、无索引查询）
- [ ] 一致性？（遵循项目约定？）

### Deep（1-2 h）

Standard 全部 + 扩展：

- [ ] 架构合理？（职责清晰、依赖方向正确）
- [ ] 线程 / 并发安全？
- [ ] 事务边界正确？
- [ ] 向后兼容？（API、数据库、配置）
- [ ] **派发 `opc-security-auditor`** 做 OWASP 完整审查
- [ ] 性能基准测试？
- [ ] 文档更新？
- [ ] 边界条件全覆盖？

### 逐文件审查顺序（从高风险到低风险）

1. 数据库迁移 / 模型变更
2. API 路由 / 控制器
3. 业务逻辑 Service
4. UI 组件
5. 测试文件
6. 配置 / 工具文件

### 审查礼仪

- 指出问题，也认可优点
- 提供具体修复建议（不只说"这不对"）
- 区分"必须"和"建议"
- 代码示例 > 文字描述

### 一人公司审查策略

- 没有同事 → AI 是你的审查者
- 每个阶段 / 计划完成后自动触发 **Standard** 审查
- 支付 / 认证相关变更必须 **Deep** 审查
- 审查报告写入 `SUMMARY.md` 或 PR 描述
