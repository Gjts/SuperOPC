# Verification Patterns — 验证模式

如何验证不同类型的产物是真实实现而非桩代码。

## 核心原则

**存在 ≠ 实现**

文件存在不代表功能正常。验证必须检查：
1. **Exists** — 文件在预期路径
2. **Substantive** — 内容是真实实现，不是占位符
3. **Wired** — 连接到系统的其余部分
4. **Functional** — 调用时实际工作

1-3 可以程序化检查。4 通常需要人工验证。

## 桩代码通用模式

这些模式表明是占位符代码：

**注释型桩：**
- `TODO` / `FIXME` / `XXX` / `HACK` / `PLACEHOLDER`
- `implement` / `add later` / `coming soon`

**空实现：**
- `return null` / `return undefined` / `return {}` / `return []`
- `pass` / `...` / `nothing`
- 仅 `console.log` 的函数

**硬编码值：**
- 硬编码 ID、计数或内容（应该是动态的）
- `"Test User"` / `"test@example.com"` 在生产代码中

## 快速验证清单

### 组件清单
- [ ] 文件在预期路径存在
- [ ] 导出函数/组件
- [ ] 返回 JSX（非 null/空）
- [ ] 无占位文本
- [ ] 使用 props 或 state
- [ ] 事件处理器有真实实现
- [ ] 在应用中被使用

### API Route 清单
- [ ] 文件在预期路径存在
- [ ] 导出 HTTP 方法处理器
- [ ] 处理器超过 5 行
- [ ] 查询数据库或服务
- [ ] 返回有意义的响应
- [ ] 有错误处理
- [ ] 验证输入
- [ ] 被前端调用

### 接线验证
- [ ] 组件 → API：fetch/axios 调用存在且使用响应
- [ ] API → 数据库：查询存在且结果被返回
- [ ] 表单 → 处理器：onSubmit 调用 API/mutation
- [ ] 状态 → 渲染：状态变量出现在 JSX 中

## 何时需要人工验证

**始终需要人工：**
- 视觉外观
- 用户流程完成度
- 实时行为（WebSocket, SSE）
- 外部服务集成（Stripe, 邮件发送）
- 错误消息清晰度
- 性能感知

## 人工验证请求格式

```markdown
## 需要人工验证

### 1. 聊天消息发送
**测试：** 输入消息并点击发送
**预期：** 消息出现在列表中，输入框清空
**检查：** 刷新后消息是否持久？

### 2. 错误处理
**测试：** 断网后尝试发送
**预期：** 显示错误消息，消息不丢失
**检查：** 重连后能否重试？
```
