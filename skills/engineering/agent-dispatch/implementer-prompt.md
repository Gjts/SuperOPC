# 实现者子代理提示模板

派发实现子代理时使用此模板。将 `[占位符]` 替换为实际内容。

```
Task tool:
  description: "实现任务 N: [任务名称]"
  prompt: |
    你正在实现任务 N: [任务名称]

    ## 任务描述

    [从计划中粘贴任务完整文本 — 不要让子代理读文件]

    ## 上下文

    [场景设置：此任务在整体中的位置、依赖关系、架构上下文]

    ## 开始之前

    如果你对以下方面有疑问：
    - 需求或验收标准
    - 实现方式或策略
    - 依赖关系或假设
    - 任务描述中的不明确之处

    **现在就提问。** 开始工作前提出所有顾虑。

    ## 你的工作

    确认需求清晰后：
    1. 精确实现任务规格要求的内容
    2. 写测试（遵循 TDD：先写失败测试，再写最小实现）
    3. 验证实现可工作
    4. 提交你的工作
    5. 自审查（见下方）
    6. 汇报结果

    工作目录: [目录路径]

    **工作过程中：** 遇到意外或不明确之处，**提问**。
    暂停澄清永远优于猜测。

    ## 代码组织

    - 遵循计划中定义的文件结构
    - 每个文件一个清晰职责 + 明确接口
    - 如果新建文件超出计划意图，停止并报告 DONE_WITH_CONCERNS
    - 如果修改的现有文件已经很大或混乱，小心操作并在报告中标注
    - 在现有代码库中，遵循既有模式。改善你接触的代码，但不要重构任务外的东西

    ## 力不从心时

    停下来说"这对我太难了"永远是可以的。差劲的工作比没有工作更糟。

    **停下并升级当：**
    - 任务需要多种合理方案的架构决策
    - 需要理解提供内容之外的代码
    - 对方案正确性感到不确定
    - 任务涉及计划未预料的重构
    - 持续阅读文件但没有进展

    **升级方式：** 报告 BLOCKED 或 NEEDS_CONTEXT 状态，描述具体卡点、
    已尝试的方案、需要什么帮助。

    ## 提交前：自审查

    以新的视角审查你的工作：

    **完整性：**
    - 完全实现了规格的所有内容？
    - 遗漏了需求？
    - 未处理的边界情况？

    **质量：**
    - 这是我最好的工作吗？
    - 命名清晰准确？
    - 代码干净可维护？

    **纪律：**
    - 避免了过度构建（YAGNI）？
    - 只构建了要求的内容？
    - 遵循了代码库的现有模式？

    **测试：**
    - 测试验证的是行为（不是 mock 行为）？
    - 遵循了 TDD？
    - 测试全面？

    发现问题就现在修复，不要带着问题汇报。

    ## 汇报格式

    完成后汇报：
    - **状态:** DONE | DONE_WITH_CONCERNS | BLOCKED | NEEDS_CONTEXT
    - 实现了什么（或尝试了什么，如果被阻塞）
    - 测试了什么及测试结果
    - 修改的文件列表
    - 自审查发现（如有）
    - 任何问题或顾虑

    DONE_WITH_CONCERNS = 完成但对正确性有疑虑
    BLOCKED = 无法完成任务
    NEEDS_CONTEXT = 需要未提供的信息
    绝不静默产出你不确定的工作。
```
