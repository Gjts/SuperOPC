## 验证循环

**这是刚性技能。不可跳过验证。**

**宣布：** "我正在使用 verification-loop 技能验证交付物。"

## 何时激活

- 阶段/计划执行完成后
- 代码审查前的自检
- 里程碑发布前
- 用户报告"某功能不工作"

## 四层验证

### 层 1: 存在性验证

```
问题：承诺的文件/功能是否存在？

检查：
- [ ] 计划中列出的每个文件都已创建？
- [ ] 路由/端点已注册？
- [ ] 数据库迁移已应用？
- [ ] 环境变量已配置？
```

### 层 2: 实质性验证

```
问题：文件/功能是否有真实内容？（不是空壳）

检查：
- [ ] 组件渲染了真实内容（不是 placeholder）？
- [ ] API 端点返回真实数据（不是 TODO）？
- [ ] 测试验证了真实行为（不是 test.skip）？
- [ ] 错误处理有具体逻辑（不是空 catch）？
```

### 层 3: 接线验证

```
问题：组件之间正确连接了吗？

检查：
- [ ] 前端调用了正确的 API 端点？
- [ ] API 路由连接到正确的 Service？
- [ ] Service 使用正确的 Repository 方法？
- [ ] 数据库表有正确的外键关系？
```

### 层 4: 功能验证

```
问题：功能按预期工作吗？

检查：
- [ ] 所有测试通过？
- [ ] 手动测试关键用户路径？
- [ ] 边界条件处理正确？
- [ ] 错误场景显示正确信息？
```

## Nyquist 验证（采样检查）

当计划产出太多无法逐一检查时，按 Nyquist 定理采样：

```
采样率 = 文件总数 的 2x 关键频率

实际操作：
- < 10 文件：全部检查
- 10-30 文件：检查 50%（所有关键路径 + 随机采样）
- 30+ 文件：检查 30%（所有关键路径 + 分层随机采样）
```

### 采样优先级

1. **必查**：入口文件、API 路由、数据库模型、认证逻辑
2. **高优**：业务逻辑 Service、关键 UI 组件
3. **采样**：工具函数、配置文件、样式

## 节点修复（Node Repair）

验证发现问题时的自动修复流程：

```
发现问题 → 分类严重度
  ├── 🔴 Critical: 功能完全不工作 → 立即修复
  ├── 🟠 High: 功能部分工作 → 当前会话修复
  ├── 🟡 Medium: 边界条件/样式 → 记录 TODO
  └── 🔵 Low: 代码质量/命名 → 记录 TODO

修复预算：最多 2 次重试
  ├── 重试 1: 尝试最小修复
  ├── 重试 2: 扩大范围修复
  └── 仍然失败 → 升级为阻塞，记录到 STATE.md
```

## 验证报告格式

```markdown
## 验证报告

**计划：** [phase]-[plan]
**日期：** [YYYY-MM-DD]

### 结果摘要
- 存在性：✅ 8/8
- 实质性：✅ 7/8（1 个 placeholder 发现）
- 接线：✅ 8/8
- 功能：✅ 6/8（2 个边界条件问题）

### 发现
1. [MEDIUM] `UserCard` 空状态显示空白而非提示
2. [LOW] `fetchUsers` 缺少超时处理

### 修复
1. ✅ 已修复（节点修复 #1）
2. 📝 记录到 TODO
```

## 一人公司验证策略

- **每个计划完成后必须验证**（不是"有空再检查"）
- 关键路径全查，其余 Nyquist 采样
- 修复预算严格（2 次），避免无限循环
- 验证报告写入 SUMMARY.md
