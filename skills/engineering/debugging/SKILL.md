## 系统化调试

**这是刚性技能。严格遵循，不可跳过。**

**宣布：** "我正在使用 debugging 技能，先找根因再修复。"

## 铁律

```
没有根因调查，就不修复。
```

## 四阶段强制门控

你**必须**完成每个阶段后才能进入下一个。

### 阶段 1: 根因调查

**在尝试任何修复之前：**

1. **仔细阅读错误信息**
   - 不要跳过错误或警告
   - 错误信息通常包含确切的解决方案
   - 阅读完整的堆栈跟踪

2. **稳定重现问题**
   - 记录确切的重现步骤
   - 确认问题是否 100% 可重现
   - 如果间歇性出现，记录频率和条件

3. **检查最近的变更**
   - `git log --oneline -10`
   - `git diff HEAD~3`
   - 最近的变更是最可能的原因

4. **追踪数据流**
   - 从错误点向上追踪
   - 找到数据在哪里偏离了预期
   - 添加日志语句来缩小范围

### 阶段 2: 模式分析

1. **查找工作示例**
   - 系统中有没有类似功能正常工作的代码？
   - 对比工作的和不工作的

2. **识别差异**
   - 工作版本和不工作版本之间的区别是什么？
   - 环境差异？数据差异？时序差异？

### 阶段 3: 假设与测试

**科学方法：**

1. **形成单一假设**
   - 明确陈述："我认为 X 是根因，因为 Y"
   - 写下来

2. **最小化测试**
   - 设计最小的实验来验证或推翻假设
   - 一次只改一个变量

3. **验证后再继续**
   - 假设成立 → 进入阶段 4
   - 假设不成立 → 回到阶段 1，带着新信息

### 阶段 4: 实施修复

1. **创建失败测试**（调用 TDD 技能）
   - 写一个重现 bug 的测试
   - 确认测试失败

2. **实施单一修复**
   - 只修复根因
   - 不要顺便重构其他东西

3. **验证修复**
   - 新测试通过
   - 所有现有测试通过
   - 手动验证问题消失

4. **修复失败处理**
   - 计数：你已经尝试了几次修复？
   - **< 3 次**：回到阶段 1，用新信息重新分析
   - **≥ 3 次**：停下来，质疑架构（见下文）

## ≥ 3 次修复失败：架构问题

如果 3 次修复都失败了，问题可能不在代码层面：
- 是否是架构设计问题？
- 是否需要重新设计这个模块？
- 是否有隐藏的前提假设是错误的？

**不要尝试第 4 次修复，除非先讨论架构。**

## 一人公司调试提示

- **记录调试过程**：下次遇到类似问题你会感谢自己
- **设置告警**：一人公司不能每天手动检查日志
- **优先修复用户可见的 bug**：没有 QA 团队帮你排优先级

## 压力测试

### 高压场景
- 线上 bug 紧急，想立刻改一堆地方碰碰运气。

### 常见偏差
- 直接猜原因，不收集证据。

### 使用技能后的纠正
- 先列假设、找证据、缩小范围，再做最小修复。
