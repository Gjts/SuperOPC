## 测试驱动开发

**这是刚性技能。严格遵循，不可跳过。**

**宣布：** "我正在使用 TDD 技能，先写测试再写代码。"

## 铁律

```
没有失败测试，就不写生产代码。
```

先写了代码？删掉它。重新来。

**没有例外：**
- 不要把它当作"参考"保留
- 不要说"我会在之后补测试"
- 不要说"这太简单了不需要测试"
- 不要说"我只是重构"

## RED-GREEN-REFACTOR 循环

### 🔴 RED：写失败测试

写一个测试，描述你**期望**代码做什么：

```bash
# 运行测试 — 必须看到它失败
npm test / pytest / cargo test / dotnet test
```

**必须验证 RED：** 亲眼看到测试失败。这证明测试真的在测试某些东西。

如果测试直接通过了，说明：
- 功能已经存在（不需要写代码）
- 测试写错了（不测试正确的东西）

### 🟢 GREEN：写最小代码

写**刚好够**让测试通过的代码。不多不少。

```bash
# 运行测试 — 必须看到它通过
npm test / pytest / cargo test / dotnet test
```

**最小 = 字面意义的最小。** 硬编码返回值都可以，如果那就够了。后续的测试会驱动你写真正的实现。

### 🔵 REFACTOR：清理代码

测试通过后，改善代码质量：
- 消除重复
- 改善命名
- 简化逻辑
- **但不改变行为** — 每次修改后运行测试确认

## 反合理化防御

**"测试后和测试先达成一样的目标"**

不。测试先回答"应该做什么？"测试后回答"做了什么？"

测试后会被你的实现偏见污染。你测试的是你构建的，而不是你应该构建的。

**"这太简单了不需要 TDD"**

简单的事情通常变得复杂。TDD 从一开始就建立安全网。如果真的简单，写测试也花不了多少时间。

**"我只是探索性编码"**

探索完了，删掉代码，用 TDD 重新写。保留"探索代码"就是在积累技术债务。

## 红旗 — 停下来，重新开始

- 代码写在测试之前
- 测试写在实现之后
- 测试直接通过（没有 RED 阶段）
- 修改测试来适配实现（而不是修改实现来适配测试）
- "我先写完再补测试"
- 跳过 REFACTOR 阶段

## 一人公司 TDD 特别提示

对一人公司来说，TDD **更加**重要，因为：
- 没有同事帮你 code review
- 6 个月后你不记得代码怎么工作
- 测试是你唯一的安全网
- 重构信心来自测试覆盖

## 压力测试

### 高压场景
- 功能看起来很简单，想先写实现再补测试。

### 常见偏差
- 把“之后补测试”当合理借口。

### 使用技能后的纠正
- 先写失败测试确认目标行为，再写最小代码通过。
