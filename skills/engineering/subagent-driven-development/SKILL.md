# 子代理驱动开发 — 双阶段审查执行引擎

每任务派发新鲜子代理 + 规格审查 + 代码质量审查 = 高质量快速迭代。

**宣布：** "我正在使用子代理驱动开发技能，每个任务由独立子代理执行并经过双阶段审查。"

## 刚性规则

1. **新鲜子代理** — 每个任务派发新子代理，不继承会话上下文
2. **双阶段审查** — 先规格合规，再代码质量，两者都通过才标记完成
3. **规格优先** — 规格审查未通过前，禁止启动代码质量审查
4. **编排器不实现** — 编排器只做上下文准备和派发，实际代码由子代理写
5. **串行执行** — 同时只有一个实现子代理工作，避免冲突
6. **绝不跳审** — 自审查不替代正式审查，两者都需要

## 适用条件

```
有实现计划? ─是─→ 任务大多独立? ─是─→ 子代理驱动开发
    │                    │
    └─否→ 先用 planning   └─否(紧密耦合)→ 手动执行或 /opc-build
```

**vs. parallel-agents (波次并行)**:
- 子代理驱动 = 单会话内串行任务 + 双阶段审查
- 波次并行 = 多会话并行任务 + 依赖验证
- 任务独立且需要审查质量 → 子代理驱动
- 任务多且需要速度 → 波次并行

## 执行流程

### Phase 1: 准备

```
1. 读取计划文件，提取全部任务（含完整文本）
2. 记录每个任务的上下文和依赖
3. 创建 TodoWrite 追踪全部任务
4. 确认在独立分支上工作（非 main/master）
```

### Phase 2: 逐任务执行

```
for each task:
  ┌─────────────────────────────────────────────────────┐
  │ 1. 派发实现者子代理 (./implementer-prompt.md)       │
  │    ↓                                                 │
  │ 2. 处理实现者状态:                                   │
  │    - DONE → 进入审查                                 │
  │    - DONE_WITH_CONCERNS → 评估后进入审查             │
  │    - NEEDS_CONTEXT → 补充上下文后重新派发            │
  │    - BLOCKED → 升级处理（见下方）                    │
  │    ↓                                                 │
  │ 3. 派发规格审查子代理 (./spec-reviewer-prompt.md)    │
  │    - ✅ 通过 → 继续                                  │
  │    - ❌ 问题 → 实现者修复 → 重新审查（循环）         │
  │    ↓                                                 │
  │ 4. 派发代码质量审查子代理                            │
  │    (./code-quality-reviewer-prompt.md)                │
  │    - ✅ 通过 → 标记完成                              │
  │    - ❌ 问题 → 实现者修复 → 重新审查（循环）         │
  │    ↓                                                 │
  │ 5. 标记任务完成                                      │
  └─────────────────────────────────────────────────────┘
```

### Phase 3: 收尾

```
1. 全部任务完成后，派发最终代码审查子代理（整体实现审查）
2. 使用 ship 工作流完成分支
```

## 模型选择策略

用能完成任务的最轻模型，节省成本加速迭代：

| 任务类型 | 推荐模型 | 信号 |
|---------|---------|------|
| 机械实现 | 快速/低成本 | 1-2文件、明确规格、无设计判断 |
| 集成任务 | 标准 | 多文件协调、模式匹配 |
| 架构/审查 | 最强可用 | 设计判断、全局理解 |

## 实现者状态处理

| 状态 | 含义 | 行动 |
|------|------|------|
| **DONE** | 完成 | 进入规格审查 |
| **DONE_WITH_CONCERNS** | 完成但有疑虑 | 评估疑虑：正确性/范围问题先解决，观察性问题记录后审查 |
| **NEEDS_CONTEXT** | 缺少信息 | 补充上下文后重新派发 |
| **BLOCKED** | 无法完成 | 升级处理（见下方） |

### BLOCKED 升级协议

```
1. 上下文不足 → 补充上下文，同模型重试
2. 推理能力不足 → 换更强模型重试
3. 任务过大 → 拆分为子任务
4. 计划本身有问题 → 上报用户
```

**绝不**忽略升级或让同模型无改变地重试。

## 红线

**绝不：**
- 在 main/master 分支上实现（除非用户明确同意）
- 跳过任一审查阶段
- 带未修复问题继续
- 并行派发多个实现子代理
- 让子代理自己读计划文件（提供完整文本）
- 跳过场景上下文（子代理需要理解任务在整体中的位置）
- 忽略子代理的问题
- 在规格审查未通过时启动代码质量审查
- 审查发现问题后不重新审查

## 关联技能

- **tdd** — 子代理实现时遵循 TDD
- **parallel-agents** — 需要并行速度时使用波次并行替代
- **planning** — 创建本技能执行的计划
- **git-worktrees** — 设置隔离工作空间

## 压力测试

### 高压场景
- 任务多，想跳过审查加速完成。

### 常见偏差
- 跳过规格审查直接做代码质量审查。

### 使用技能后的纠正
- 始终双阶段：先规格合规（正确性），再代码质量（可维护性）。
