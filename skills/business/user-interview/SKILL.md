## 用户访谈

**核心原则：** 不问“你会不会用”，只问真实过去发生了什么。访谈要收集行为证据，不收集礼貌性鼓励。

**宣布：** "我正在使用 user-interview 技能设计和分析用户访谈。"

## The Mom Test 规则

### 应该问

- 上一次你怎么处理这个问题？
- 这件事多久发生一次？
- 最烦的部分是什么？
- 你现在在用什么替代方案？
- 为此花过多少钱/时间？

### 不该问

- 你觉得这个想法怎么样？
- 如果我做了你会用吗？
- 你会不会付费？
- 这个功能是不是很棒？

## 访谈流程

### 1. 选人

- 最近 30 天内实际遇到问题的人
- 能清楚描述当前流程的人
- 尽量覆盖不同成熟度，而不是全找朋友

### 2. 访谈结构

1. 背景和角色
2. 最近一次真实场景
3. 当前流程与替代方案
4. 成本与后果
5. 现有工具缺口
6. 引入解决方案后的购买流程

### 3. 访谈后立刻整理

- 真实原话
- 证据等级
- 高价值痛点
- 购买信号
- 新假设 / 被推翻假设

## 证据等级

| 等级 | 说明 |
|------|------|
| A | 描述真实过去行为，有成本或频率 |
| B | 描述当前流程，但证据不完整 |
| C | 观点或猜测 |
| D | 纯鼓励、礼貌性反馈 |

优先用 A/B 级证据做决策。

## 输出格式

```markdown
## 访谈摘要
- 受访者：
- 角色：
- 场景：

### 证据
- [A] 上次处理用了 3 小时
- [B] 现有工具无法协作

### 痛点
- 痛点 1
- 痛点 2

### 下一步
- 验证假设 X
- 调整 landing page 文案
```

## 反模式

- 讲你的产品太多，用户讲得太少
- 一开始就 demo
- 把意见当事实
- 访谈后不做结构化整理

## 输出

1. 访谈提纲
2. 证据分级摘要
3. 高价值痛点列表
4. 对产品/文案/定价的调整建议
