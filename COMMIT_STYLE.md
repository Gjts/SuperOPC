# COMMIT_STYLE.md

SuperOPC 仓库提交信息规范。

本规范用于统一项目的 `git commit` 风格，确保版本历史满足以下目标：

- 可快速回顾每个版本的**目标**
- 可追踪每次提交影响的**具体文件**
- 可识别每个版本融合了哪些**来源项目/方法论**
- 可作为 README / CHANGELOG / ROADMAP 之外的**精细历史索引**

---

## 1. 适用范围

本规范适用于：

- 所有功能性提交
- 所有结构性重组提交
- 所有技能、代理、命令、规则、钩子、模板、文档类提交
- 所有希望保留中长期历史价值的提交

建议以下提交**严格遵循本规范**：

- `feat`
- `refactor`
- `docs`
- `perf`
- `build`
- `chore`（当提交内容涉及系统结构、工具链、规范、模板时）

以下提交可以适度简化，但仍建议尽量保留结构：

- `fix`
- `test`
- `style`

---

## 2. 总体原则

每个 commit message 应同时满足：

1. **标题可扫描**
   - 一眼看出本次提交的核心主题
2. **正文可审计**
   - 能定位到具体文件和变更意图
3. **来源可追踪**
   - 能看出哪些内容来自 ECC / GSD / Superpowers / 其他来源
4. **分类可浏览**
   - 文件较多时，允许按模块/能力/目录进行分类分组
5. **信息不过载**
   - 重点描述“新增/修改了什么、为什么、关键参数是什么”
   - 不要求把代码实现细节逐行搬进 commit body

---

## 3. 标题格式

采用 Conventional Commits 风格：

```/dev/null/commit-title-format.txt#L1-3
<type>(<scope>): <summary>
<type>: <summary>
Initial commit
```

### 推荐类型

- `feat`：新增能力
- `fix`：修复问题
- `docs`：文档更新
- `refactor`：重构
- `perf`：性能优化
- `test`：测试补充或重构
- `build`：构建、转换、发布、依赖相关
- `chore`：杂项维护
- `style`：格式与风格，不改逻辑

### 示例

```/dev/null/commit-title-examples.txt#L1-8
feat(skills): 工程技能深化至 19 个（新增 15 个全栈 + 技术栈专属技能）
feat(state): 状态管理 + 文件系统 + 模板系统（v0.3.0 补完 + v0.4.0 完整实现）
feat(agents): 代理扩展至 15 个 + 波次并行执行引擎
docs(readme): 更新架构图与安装说明
refactor(commands): 统一命令目录结构与命名规范
build(convert): 扩展多工具格式转换输出
fix(hooks): 修复状态文件锁超时判断错误
```

## 4. 正文固定三段式

除极小提交外，正文必须包含以下三个一级段落，顺序固定：

1. `概述：`
2. `逐文件详细说明：`
3. `来源引用：`

标准骨架如下：

```/dev/null/commit-body-template.txt#L1-18
<type>(<scope>): <title>

概述：
- 版本目标：
- 本次改动解决的问题：
- 来源融合说明：

逐文件详细说明：
【分类 1（数量 + 来源/适配说明）】
- path/to/file1 — 功能描述：
  - 关键配置项/参数：
- path/to/file2 — 功能描述：
  - 关键配置项/参数：

【分类 2（数量 + 来源/适配说明）】
- path/to/file3 — 功能描述：
  - 关键配置项/参数：

来源引用：
- ECC：
  - 具体借鉴内容
- GSD：
  - 具体借鉴内容
- Superpowers：
  - 具体借鉴内容
```

---

## 5. 第一段：概述

`概述：` 用来回答三件事：

1. **这个版本的目标是什么**
2. **这次提交解决了什么问题**
3. **融合了哪些来源的什么思路**

推荐写成 2-4 个项目符号。

### 推荐写法

```/dev/null/overview-example.txt#L1-5
概述：
- 完成 SuperOPC v0.5.0 阶段一（基础）最后一个版本，将 skills/engineering/ 从 4 个扩展到 19 个。
- 本版本目标是补齐一人公司场景下的全栈工程技能库，使系统从基础工程辅助升级为可直接指导交付的工程知识层。
- 本次提交解决了工程技能覆盖不足、技术栈专属模式缺失、验证与交付模式不系统的问题。
- 来源融合说明：融合 ECC skills/ 的工程实践、GSD 的验证与门控模式，以及 Superpowers 的技能优先与执行闭环理念。
```

### 要点

- 可以出现版本号，如 `v0.4.0`、`v0.5.0`
- 可以说明“这是阶段收尾 / 补完项 / 扩展项”
- 如果是纯维护提交，也要说明为什么值得进入历史

---

## 6. 第二段：逐文件详细说明

这是本规范的核心。

要求：

- 必须写出**完整文件路径**
- 必须写出**功能描述**
- 如有配置、阈值、模式、状态值、命令参数等，必须写出**关键配置项/参数**
- 文件较多时，**允许按分类分组**
- 分类不会替代逐文件说明，分类只是辅助阅读

### 6.1 分类标题格式

统一采用：

```/dev/null/category-format.txt#L1-3
【分类名（数量 + 来源/适配说明）】
- `完整文件路径` — 功能描述：
  - 关键配置项/参数：
```

### 6.2 推荐分类标题示例

```/dev/null/category-examples.txt#L1-12
【通用工程技能（11 个，源自 ECC skills/ 适配一人公司场景）】
【技术栈专属技能（4 个，匹配用户技术栈）】
【新增专业代理（9 个，扩展任务处理边界）】
【代理编排与执行能力（1 个，增强波次并行与失败隔离）】
【Hooks 运行时门控（10 个，编辑/提交/会话阶段质量保障）】
【通用编码规则（5 个，适用于所有语言）】
【语言专属规则（6 个，匹配 TypeScript / C# / Python / Kotlin 技术栈）】
【引用文档（6 个，源自 GSD references 模式）】
【状态管理增强（2 个，服务 .opc/ 文件级状态系统）】
【配置与模板系统（8 个，支撑规划/执行/总结工作流）】
【项目文档更新（5 个，反映版本状态与使用方式）】
【系统说明同步（2 个，更新代理/技能/规则索引）】
```

### 6.3 单文件条目格式

统一采用：

```/dev/null/file-item-format.txt#L1-3
- `完整文件路径` — 功能描述：
  - 关键配置项/参数：
```

同组多个文件时，必须每个文件单独换行，不要在同一条目中串联多个文件路径。

例如：

```/dev/null/file-item-format-example.txt#L1-6
- `scripts/hooks/state_file_lock.py` — STATE 文件锁：
  - 关键配置项/参数：锁文件 `.opc/STATE.md.lock`；30 秒超时；建议性守卫；兼容 `.planning/`。
- `templates/config.json` — 默认项目配置模板：
  - 关键配置项/参数：mode、granularity、model_profile、workflow、parallelization、git、gates、safety、hooks。
```

### 6.4 多文件合并描述的条件

仅在以下情况下允许将多个文件合并为一个条目：

- 文件是同一组规则或模板，结构完全一致
- 功能属于同一能力集合，拆开写只会造成重复
- 合并后仍能明确知道涉及了哪些文件

例如：

```/dev/null/file-group-example.txt#L1-8
- `rules/python/coding-style.md` — Python 编码规范：
  - 关键配置项/参数：PEP 8、Black、类型提示、文件 < 400 行、函数 < 50 行。
- `rules/python/testing.md` — Python 测试规范：
  - 关键配置项/参数：pytest、pytest-asyncio、pytest-cov、覆盖率 ≥ 80%。
- `rules/python/security.md` — Python 安全规范：
  - 关键配置项/参数：参数化查询、Pydantic 输入验证、环境变量密钥管理、pip-audit。
```

### 6.5 推荐内容粒度

每个条目尽量说明：

- 这个文件是什么
- 它在系统里承担什么作用
- 关键参数/阈值/状态值/命令选项/策略是什么

不要求说明：

- 所有实现细节
- 所有内部函数
- 所有格式化变更

---

## 7. 第三段：来源引用

`来源引用：` 不是简单写名字，而是要尽量回答：

- 借鉴了哪个来源
- 借鉴了其中什么内容
- 落在本仓库的哪些目录/文件/模式上

### 推荐最少包含

- `ECC`
- `GSD`
- `Superpowers`

如果还有其他来源，继续追加：

- `Minimalist Entrepreneur`
- `last30days`
- `Follow Builders`
- `Skill From Masters`
- `Agency-Agents`
- `Claude Code Best Practice`
- 其他你明确参考的来源

### 推荐写法

```/dev/null/source-reference-example.txt#L1-12
来源引用：
- ECC：
  - `skills/engineering/api-design/`、`architecture-decision-records/`、`backend-patterns/` 等工程技能主题与结构化写法；
  - hooks / rules 的目录组织方式；
  - 工程实践文档化方法。
- GSD：
  - 四层验证与 gates 门控思路；
  - 多代理编排、波次执行、状态化工作流参考；
  - references/ 风格的参考文档组织方式。
- Superpowers：
  - 技能优先的系统组织方式；
  - TDD、调试、执行闭环与工程导向实践。
```

### 反例

不推荐只写：

```/dev/null/source-reference-bad-example.txt#L1-4
来源引用：
- ECC
- GSD
- Superpowers
```

问题在于：**没有说明具体借鉴了什么。**

## 8. 何时允许简化

以下情况可以适度简化正文，但仍建议保留三段式：

### 8.1 小型修复提交

例如只修 1 个 bug、只改 1-2 个文件。

可简化为：

```/dev/null/small-fix-template.txt#L1-13
fix(hooks): 修复状态文件锁超时判断错误

概述：
- 修复状态文件锁在旧锁清理场景下判断不准确的问题。
- 本次改动解决了并行执行时锁误判导致的重复警告。
- 来源融合说明：延续 GSD 的状态化工作流和 ECC 的 hook 守卫思路。

逐文件详细说明：
- scripts/hooks/state_file_lock.py — 修复锁超时判断逻辑：
  - 关键配置项/参数：保留 30 秒超时策略，修正旧锁比较逻辑。

来源引用：
- ECC：hook 守卫和建议性保护模式。
- GSD：状态文件与并行执行协同思路。
```

### 8.2 初始提交

初始提交没有具体文件时，可适度说明“仓库初始化基线”。  
但如果初始提交本身已经包含实际内容，仍应按正常规范写。

## 11. 反例

### 11.1 过短，无法追踪

```/dev/null/bad-example-too-short.txt#L1-2
feat: update skills
add more engineering skills
```

问题：

- 看不出版本目标
- 看不出具体文件
- 看不出来源融合
- 历史价值很低

### 11.2 只有分类，没有逐文件

```/dev/null/bad-example-no-files.txt#L1-8
feat(skills): 扩展工程技能

概述：
- 新增很多工程技能。

逐文件详细说明：
【通用工程技能（11 个）】
【技术栈专属技能（4 个）】
```

问题：

- 分类存在，但没有文件路径
- 无法定位具体变更

### 11.3 只有来源名，没有具体内容

```/dev/null/bad-example-no-detail.txt#L1-7
来源引用：
- ECC
- GSD
- Superpowers
```

问题：

- 没说明具体借鉴内容
- 难以进行历史追踪与来源审计

## 12. 实操建议

### 12.1 提交前先整理这三件事

在写 commit message 前，先明确：

1. **本次版本目标是什么**
2. **具体改了哪些文件**
3. **每一组文件分别来源于哪种模式或参考**

### 12.2 文件较多时的推荐顺序

优先按下面顺序组织正文：

1. 核心能力文件
2. 支撑性配置/模板
3. 文档同步文件

### 12.3 不要把正文写成变更日志复制品

commit body 要回答的是：

- 为什么做
- 改了哪里
- 关键配置是什么
- 来源是什么

而不是机械列出所有 diff 行数。

---

## 13. 推荐工作流

建议在提交前使用下面的思路准备 commit body：

```/dev/null/commit-workflow.txt#L1-8
1. 先写标题
2. 写概述（目标 / 问题 / 来源融合）
3. 列出涉及文件
4. 按分类分组
5. 为每个文件补功能描述
6. 提炼关键配置项/参数
7. 补来源引用
8. 最后检查是否满足三段式
```

---

## 14. 最终检查清单

提交前自查：

- [ ] 标题符合 Conventional Commits 风格
- [ ] 正文包含 `概述：`
- [ ] 正文包含 `逐文件详细说明：`
- [ ] 正文包含 `来源引用：`
- [ ] 每个文件条目包含完整文件路径
- [ ] 同组多个文件时每个文件单独换行
- [ ] 每个文件条目包含功能描述
- [ ] 关键配置项/参数已写出
- [ ] 文件较多时已按分类分组
- [ ] 分类标题使用 `【分类名（数量 + 来源/适配说明）】`
- [ ] 来源引用写到了具体内容层级，而不是只写来源名
- [ ] 提交信息能让未来的你在 30 秒内理解这次提交做了什么

---

## 15. 一句话规范

> SuperOPC 的 commit message 采用“**标题 + 三段式正文**”规范：`概述` 说明目标与来源融合，`逐文件详细说明` 支持按分类分组展开并要求写出完整文件路径且每个文件单独换行，`来源引用` 明确标注具体借鉴内容。
