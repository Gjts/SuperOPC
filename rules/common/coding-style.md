# Coding Style — SuperOPC 通用编码规范

> 适用于所有语言。语言特定规则在对应子目录中扩展。

## 不可变性（关键）

始终创建新对象，不要修改原有对象：

```
错误：modify(original, field, value) → 原地修改
正确：update(original, field, value) → 返回新副本
```

## 核心原则

### KISS
- 选择最简单的可行方案
- 避免过早优化
- 优先清晰度，而非巧妙性

### DRY
- 重复逻辑提取为共享函数
- 避免复制粘贴导致的实现漂移
- 当重复是真实的（而非推测的）时引入抽象

### YAGNI
- 不在需要之前构建功能或抽象
- 先简单实现，在压力真实时再重构

## 文件组织

**多个小文件 > 少数大文件**
- 高内聚，低耦合
- 典型 200-400 行，**最大 800 行**
- 按功能/领域组织，而非按类型
- 任何方法不超过 200 行，嵌套不超过 5 层

## 错误处理

- 每层显式处理错误
- UI 层提供用户友好的错误消息
- 服务端记录详细错误上下文
- 绝不静默吞掉错误

## 输入验证

- 在系统边界处验证所有输入
- 优先使用 schema 验证（Zod / FluentValidation / Pydantic）
- 快速失败 + 清晰错误消息
- 永不信任外部数据

## 命名约定

| 类型 | 风格 | 前缀 |
|------|------|------|
| 变量/函数 | camelCase | — |
| 布尔值 | camelCase | is/has/should/can |
| 类型/接口/组件 | PascalCase | — |
| 常量 | UPPER_SNAKE_CASE | — |
| 自定义 Hook | camelCase | use |

## 代码坏味道

- **深层嵌套** → 使用早返回 / 卫语句
- **魔法数字** → 使用命名常量
- **超长函数** → 拆分为单一职责的小函数

## 代码质量清单

完成前检查：
- [ ] 代码可读、命名清晰
- [ ] 函数小（<50 行）
- [ ] 文件聚焦（<800 行）
- [ ] 无深层嵌套（>4 层）
- [ ] 正确的错误处理
- [ ] 无硬编码值（使用常量或配置）
- [ ] 无可变修改（使用不可变模式）
