# Common Patterns — SuperOPC 通用设计模式

## 骨架项目模式

实现新功能时：
1. 搜索经过实战检验的骨架项目
2. 使用并行代理评估选项：安全评估 + 可扩展性分析 + 相关性评分
3. 克隆最佳匹配作为基础
4. 在已验证的结构上迭代

## Repository 模式

封装数据访问在一致的接口后面：
- 定义标准操作：findAll, findById, create, update, delete
- 具体实现处理存储细节（数据库、API、文件）
- 业务逻辑依赖抽象接口，不依赖存储机制
- 便于替换数据源 + 简化测试

## API 响应格式

所有 API 响应使用统一信封：

```json
{
  "success": true,
  "data": { ... },
  "error": null,
  "meta": { "total": 100, "page": 1, "limit": 20 }
}
```

- 包含 success/status 指示器
- data 字段（错误时为 null）
- error 字段（成功时为 null）
- 分页响应包含 meta

## 错误处理模式

### 错误边界（前端）
- 组件级错误边界捕获渲染错误
- 全局错误边界作为最后防线
- 每个边界提供有意义的回退 UI

### Result 模式（后端）
- 返回 Result<T, Error> 而非抛异常（适用时）
- 使错误路径显式可见
- 避免异常作为流程控制

## 一人公司模式

### 先手动后自动
1. 手动完成流程 10 次
2. 找出重复和瓶颈
3. 只自动化真正重复的部分

### 最小化依赖
- 每增加一个依赖 = 一个维护承诺
- 能用标准库就不引入第三方
- 选择维护活跃、社区大的依赖
